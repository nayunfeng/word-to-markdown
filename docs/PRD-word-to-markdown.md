# Word 转 Markdown 工具 PRD

## 1. 文档信息

- 项目名称：Word 转 Markdown 工具
- 文档类型：产品需求文档（PRD）
- 当前阶段：需求阶段
- 项目目标：将企业内部 Word `.docx` 文档转换为结构化 Markdown，用于文档迁移、知识库入库与 AI 检索预处理。

---

## 2. 项目背景

企业内部存在大量 Word 文档，常见用途包括：

- 接口说明书
- 制度文档
- 操作手册
- 项目方案
- 合同附件
- 流程文档

这些文档通常存在以下问题：

1. 内容分散，难以统一归档
2. 原始 Word 文档不便直接进入知识库
3. AI 检索或向量化前需要结构化清洗
4. 图文、表格、章节等信息若处理不当，后续利用价值会明显下降

因此需要一个工具，将 Word 文档中的章节、段落、列表、表格、图片等结构提取出来，输出为结构化 Markdown，作为后续知识库或 AI 处理入口。

---

## 3. 项目目标

### 3.1 目标定义

本项目不是做 Word 样式的 1:1 还原，而是做：

**企业 Word 文档 → 结构化 Markdown → 知识库 / AI 预处理入口**

### 3.2 目标场景

工具产出结果可用于：

- 文档迁移到 Git / Wiki / Markdown 文档库
- 作为知识库原始数据
- 作为本地模型 / 向量库的切分前处理结果
- 批量归档和统一检索

---

## 4. 适用范围

### 4.1 支持范围

- 支持 `.docx` 文档
- 支持单文件处理
- 支持目录批量处理
- 支持递归扫描子目录中的 `.docx` 文件

### 4.2 不支持范围

- 不支持老式 `.doc` 文档直接解析
- 不追求 Word 样式完全还原
- 不作为富文本编辑器使用

---

## 5. 用户角色

### 5.1 主要用户

1. 文档整理人员
2. 知识库维护人员
3. AI / RAG 数据预处理人员
4. 开发 / 测试 / 运维文档归档人员

### 5.2 用户目标

- 快速把 Word 变成 Markdown
- 保留章节结构
- 尽量保留表格与图片信息
- 支持批量转换，减少人工复制整理

---

## 6. 需求范围

## 6.1 输入需求

系统应支持输入单个 `.docx` 文件。

系统应支持输入目录路径，并批量处理目录中的所有 `.docx` 文件。

系统应支持递归扫描子目录中的 `.docx` 文件。

系统应在输入路径不存在、文件格式错误时给出明确提示。

---

## 6.2 输出需求

系统应输出完整 Markdown 文件 `full.md`。

系统应支持按标题拆分输出多个 Markdown 文件。

系统应支持输出图片资源目录 `media/`。

系统应支持输出章节目录 `sections/`。

系统应保证输出文件命名安全，不因标题特殊字符导致文件写入失败。

### 标准输出目录示例

```text
output/
├─ full.md
├─ media/
│  ├─ image_001.png
│  └─ image_002.jpg
└─ sections/
   ├─ 001-项目背景.md
   ├─ 002-系统设计.md
   └─ 003-接口说明.md
```

---

## 6.3 文本解析需求

系统应识别普通段落并输出为 Markdown 段落。

系统应识别标准 Word Heading 样式标题。

系统应支持启发式识别常见非标准标题，例如：

- 第一章
- 一、
- （一）
- 1.
- 1.1
- （1）

系统应支持基础列表识别，包括：

- Word 列表样式
- `•`
- `-`
- `*`

系统应在 Markdown 中保留文档的逻辑顺序，保证段落、表格、图片的输出顺序与原文基本一致。

---

## 6.4 标题拆分需求

系统应支持按标题层级拆分 Markdown 文件。

系统应支持配置拆分层级，例如：

- 按一级标题拆分
- 按二级标题拆分

系统应保证拆分后的章节顺序正确。

系统应将拆分结果输出到 `sections/` 目录。

---

## 6.5 表格处理需求

系统应支持基础二维表格转 Markdown 表格。

系统应支持表格单元格中的换行文本输出。

系统应尽量保留表格在文档中的相对位置。

系统应允许复杂表格存在降级处理。

### 一期不强制完全支持的场景

- 合并单元格
- 多级表头
- 复杂跨行跨列
- 表格中深度嵌套内容

---

## 6.6 图片处理需求

系统应支持抽取 Word 文档中的嵌入图片。

系统应将图片输出到 `media/` 目录。

系统应在 Markdown 中生成图片引用。

系统应尽量按原文位置插入图片引用，而不是统一在文末堆叠。

系统应允许图片抽取失败时不中断整篇文档转换，至少保证文本和表格结果可输出。

---

## 6.7 批量处理需求

系统应支持目录批量转换。

系统应支持递归扫描子目录。

系统应按源文档分别输出对应结果目录。

批量处理时，单个文件失败不应导致全部任务中断。

系统应输出可读的处理日志，至少包含：

- 当前处理文件
- 是否成功
- 输出路径
- 失败原因

---

## 6.8 命令行使用需求

系统应提供统一命令行入口。

系统应支持以下核心参数：

- 输入路径
- 输出路径
- 是否按标题拆分
- 拆分层级
- 是否递归扫描
- 是否抽取图片

系统应支持 Windows 环境下的一键运行脚本。

---

## 7. 非功能需求

## 7.1 准确性

优先保证章节、段落、表格、图片的结构正确。

不以 Word 样式完全还原为目标。

应明确复杂文档存在一定误差空间，避免用户错误预期。

---

## 7.2 稳定性

单个文档解析失败时，应给出清晰错误提示。

批量处理时，应保证部分失败不影响整体任务继续执行。

对缺失图片、特殊表格、异常样式等情况，应尽量降级而非整体崩溃。

---

## 7.3 可维护性

系统结构应具备清晰边界，至少区分：

- 输入层
- 解析层
- Markdown 输出层
- 拆分层
- 图片处理层

对外应只保留一个推荐稳定入口，避免多版本入口导致混乱。

---

## 7.4 可扩展性

系统应支持后续能力扩展，包括但不限于：

1. 更复杂表格处理
2. 更精细的图片位置还原
3. 图注识别
4. front matter / metadata 输出
5. 知识库切分策略
6. 自动化测试与样例库建设

---

## 8. 输入输出定义

## 8.1 输入

- 单个 `.docx` 文件
- `.docx` 文件所在目录

## 8.2 输出

- `full.md`
- `sections/` 目录
- `media/` 目录
- 处理日志（控制台或文本）

---

## 9. 边界与限制

系统当前仅面向 `.docx`。

对于以下场景，允许一期存在不完全准确：

- 非标准标题识别
- 复杂表格
- 图片精确位置还原
- Word 样式保真

项目目标是结构化可用，而不是视觉完全还原。

---

## 10. 一期范围

一期建议覆盖：

1. 单文件与批量 `.docx` 处理
2. 段落、标题、列表转换
3. 基础表格转换
4. 按标题拆分
5. 图片抽取与 Markdown 引用
6. 命令行运行
7. Windows 一键执行

---

## 11. 二期规划

二期可考虑：

1. 复杂表格增强
2. 图注识别
3. front matter 输出
4. 更细粒度章节切分
5. 与知识库 / 向量库流程对接
6. 真实样例集成测试

---

## 12. 验收标准

### 12.1 基础验收

给定一个普通 `.docx` 文档，系统能够：

- 成功生成 `full.md`
- 正确输出章节结构
- 正确输出基础表格
- 成功抽取图片到 `media/`
- 按需生成 `sections/`

### 12.2 批量验收

给定一个包含多个 `.docx` 文件的目录，系统能够：

- 逐个处理文件
- 为每个文件生成对应输出目录
- 处理失败的文件能报错但不阻塞其他文件继续执行

### 12.3 使用验收

非开发人员按照说明文档，能够在 Windows 环境下完成一次完整转换。

---

## 13. 一句话结论

这是一个面向企业文档整理、知识库建设与 AI 预处理场景的 Word `.docx` 转 Markdown 工具需求，不是一个追求 Word 样式 1:1 还原的排版系统。
