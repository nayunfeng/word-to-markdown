# Word 转 Markdown 工具 PRD

## 1. 文档信息

- 项目名称：Word 转 Markdown 工具
- 文档类型：产品需求文档（PRD）
- 文档版本：v1.1
- 文档状态：待需求评审
- 当前阶段：一期需求定义
- 项目目标：将企业内部 Word `.doc` / `.docx` 文档转换为结构化 Markdown，用于文档迁移、知识库入库与 AI 检索预处理

---

## 2. 项目背景

企业内部存在大量 Word 文档，常见用途包括：

- 接口说明书
- 制度文档
- 操作手册
- 项目方案
- 合同附件
- 流程文档

这些文档通常存在以下问题：

1. 内容分散，难以统一归档
2. 原始 Word 文档不便直接进入知识库
3. AI 检索或向量化前需要结构化清洗
4. 图文、表格、章节等信息若处理不当，后续利用价值会明显下降

因此需要一个工具，将 Word 文档中的章节、段落、列表、表格、图片等结构提取出来，输出为结构化 Markdown，作为后续知识库或 AI 处理入口。

---

## 3. 项目目标与非目标

### 3.1 目标定义

本项目不是做 Word 样式的 1:1 还原，而是做：

**企业 Word 文档（`.doc` / `.docx`）→ 结构化 Markdown → 知识库 / AI 预处理入口**

### 3.2 目标场景

工具产出结果可用于：

- 文档迁移到 Git / Wiki / Markdown 文档库
- 作为知识库原始数据
- 作为本地模型 / 向量库的切分前处理结果
- 批量归档和统一检索

### 3.3 非目标

一期明确不做以下能力：

- 不直接解析老式 `.doc` 二进制格式，`.doc` 应通过预转换后进入统一解析流程
- 不追求 Word 样式、页眉页脚、分页、字体、颜色的 1:1 还原
- 不作为富文本编辑器使用
- 不保证复杂表格和复杂图文混排完全保真

---

## 4. 用户角色

### 4.1 主要用户

1. 文档整理人员
2. 知识库维护人员
3. AI / RAG 数据预处理人员
4. 开发 / 测试 / 运维文档归档人员

### 4.2 用户目标

- 快速把 Word 变成 Markdown
- 保留章节结构
- 尽量保留表格与图片信息
- 支持批量转换，减少人工复制整理

---

## 5. 范围定义

### 5.1 一期支持范围

- 支持 `.docx` 文档
- 支持 `.doc` 文档输入，并在满足依赖条件时自动预转换为 `.docx`
- 支持单文件处理
- 支持目录批量处理
- 支持递归扫描子目录中的 `.docx` / `.doc` 文件
- 支持输出完整 Markdown、章节拆分文件、图片资源与处理日志
- 支持 Windows 环境下的一键运行脚本

### 5.2 一期不支持范围

- 无转换能力依赖时的 `.doc` 直接解析
- OCR、扫描件识别
- 复杂表格的完整语义恢复
- 图注、脚注、批注、修订记录的完整结构化提取
- 在线服务、Web 页面或知识库系统直接集成

### 5.3 优先级

- Must：单文件转换、批量转换、标题/段落/列表转换、基础表格转换、图片抽取、CLI 入口、Windows 一键运行、`.doc` 预转换接入
- Should：按标题拆分、可读日志、部分失败继续执行、输出目录命名安全
- Could：front matter 输出、图注识别、更复杂表格增强

## 6. 功能需求

### 6.1 输入需求

系统应支持以下输入形式：

- 单个 `.docx` 或 `.doc` 文件路径
- 包含 `.docx` / `.doc` 文件的目录路径

系统行为要求如下：

1. 输入为文件路径时，仅处理该文件
2. 输入为目录路径时，处理目录下所有 `.docx` / `.doc` 文件
3. 开启递归参数时，应扫描所有子目录中的 `.docx` / `.doc` 文件
4. 输入路径不存在时，应立即报错并退出
5. 输入为非 `.docx` / `.doc` 文件时，应给出明确提示并退出
6. 输入目录中无可处理文件时，应给出明确提示并退出

### 6.1.1 `.doc` 兼容策略

考虑到 `.doc` 为老式二进制格式，一期不建议单独维护一套解析链路，应采用统一预转换策略：

1. 输入为 `.doc` 时，系统应先尝试转换为 `.docx`
2. 转换成功后，后续解析、拆分、图片抽取流程与 `.docx` 保持一致
3. 转换失败时，应记录明确原因并按失败文件处理
4. 批量处理场景下，单个 `.doc` 转换失败不应阻塞其他文件继续执行

### 6.1.2 `.doc` 转换依赖要求

一期允许以下任一转换方案：

1. Windows 环境下使用已安装的 Microsoft Word 进行转换
2. 使用 LibreOffice / soffice 进行无界面转换

若运行环境不具备上述依赖之一，系统应明确提示当前环境不支持 `.doc` 转换，而不是静默失败。

---

### 6.2 输出需求

系统应输出以下内容：

- 完整 Markdown 文件 `full.md`
- 按标题拆分的 `sections/` 目录
- 图片资源目录 `media/`
- 处理日志

### 6.2.1 输出目录规则

单文件模式下，单个源文档的输出目录规则为：

```text
<output_root>/<source_basename>/
```

批量模式下，每个源文档应独立输出，推荐规则为：

```text
<output_root>/<source_relative_dir>/<source_basename>/
```

例如：

```text
input/
├─ hr/制度.doc
└─ tech/接口说明.docx

output/
├─ hr/制度/
│  ├─ full.md
│  ├─ media/
│  └─ sections/
└─ tech/接口说明/
   ├─ full.md
   ├─ media/
   └─ sections/
```

### 6.2.2 命名规则

1. 输出目录名默认使用源文件名去掉扩展名后的结果
2. 标题拆分文件名格式为 `<序号>-<标题>.md`
3. 文件名中的非法字符应进行替换或移除，避免 Windows 写入失败
4. 若同目录下存在重名输出，应自动追加后缀 `_1`、`_2` 等保证不冲突
5. 重复执行同一任务时，默认覆盖该源文档对应的输出目录，不影响其他文档输出目录
6. `.doc` 转换生成的临时 `.docx` 文件不应默认保留在输出目录中，除非显式开启调试保留

### 标准输出目录示例

```text
output/
└─ 项目方案/
   ├─ full.md
   ├─ media/
   │  ├─ image_001.png
   │  └─ image_002.jpg
   └─ sections/
      ├─ 001-项目背景.md
      ├─ 002-系统设计.md
      └─ 003-接口说明.md
```

### 6.3 文本解析与标题识别需求

系统应识别普通段落并输出为 Markdown 段落。

系统应识别标准 Word Heading 样式标题。

系统应支持启发式识别常见非标准标题，例如：

- 第一章
- 一、
- （一）
- 1.
- 1.1
- （1）

### 6.3.1 标题识别优先级

标题识别优先级必须固定，避免同一段文本被不同规则反复判定：

1. Word 原生 Heading 1-6 样式
2. 明确的编号标题规则匹配
3. 其他情况按普通段落处理

### 6.3.2 启发式标题判定条件

当文本不带 Heading 样式时，仅在同时满足以下条件时才可按启发式标题处理：

1. 该段落为独立段落
2. 文本长度不超过 80 个字符
3. 文本末尾不以 `。`、`；`、`.`、`;` 结尾
4. 段落内容匹配预设标题模式

### 6.3.3 默认标题映射规则

一期默认映射规则如下：

- `Heading 1` -> Markdown `#`
- `Heading 2` -> Markdown `##`
- `Heading 3` -> Markdown `###`
- `Heading 4-6` -> 对应 Markdown `####` 到 `######`
- `第一章`、`第二章` 这类模式 -> 一级标题
- `一、`、`二、` 这类模式 -> 一级标题
- `（一）`、`（二）` 这类模式 -> 二级标题
- `1.`、`2.` 这类模式 -> 一级标题
- `1.1`、`1.2` 这类模式 -> 二级标题
- `（1）`、`（2）` 这类模式 -> 三级标题

若同一段文本同时命中多个启发式规则，应按更高层级优先。

若文本命中规则但不满足启发式标题判定条件，应按普通段落输出，避免误识别。

### 6.3.4 列表与顺序要求

系统应支持基础列表识别，包括：

- Word 列表样式
- `•`
- `-`
- `*`

系统应在 Markdown 中保留文档的逻辑顺序，保证段落、表格、图片的输出顺序与原文基本一致。

---

### 6.4 标题拆分需求

系统应支持按标题层级拆分 Markdown 文件。

系统应支持配置拆分层级，例如：

- 按一级标题拆分
- 按二级标题拆分

### 6.4.1 拆分规则

1. 开启拆分时，仍需生成完整文件 `full.md`
2. `split-level=1` 表示按一级标题生成章节文件
3. `split-level=2` 表示按二级标题生成章节文件
4. 单个章节文件应包含该标题下直到下一个同级或更高层级标题出现前的全部内容
5. 若文档中不存在目标层级标题，则不生成 `sections/`，并在日志中提示
6. 章节文件顺序必须与原文顺序一致

---

### 6.5 表格处理需求

系统应支持基础二维表格转 Markdown 表格。

系统应支持表格单元格中的换行文本输出。

系统应尽量保留表格在文档中的相对位置。

### 6.5.1 一期必须支持

- 普通二维表格
- 固定表头 + 普通数据行
- 单元格内的纯文本或换行文本

### 6.5.2 一期允许降级的场景

- 合并单元格
- 多级表头
- 复杂跨行跨列
- 表格中深度嵌套内容

### 6.5.3 降级规则

复杂表格无法稳定转换为标准 Markdown 表格时，应按以下策略降级：

1. 优先输出 HTML 表格片段到 Markdown 中
2. 若仍无法稳定输出，则输出原始文本近似内容
3. 必须在日志中记录该表格已降级处理
4. 不得因单个复杂表格导致整篇文档转换失败

---

### 6.6 图片处理需求

系统应支持抽取 Word 文档中的嵌入图片。

系统应将图片输出到 `media/` 目录。

系统应在 Markdown 中生成图片引用。

系统应尽量按原文位置插入图片引用，而不是统一在文末堆叠。

### 6.6.1 图片输出规则

1. 图片文件名默认按顺序生成为 `image_001`、`image_002` 等
2. 图片扩展名应尽量保留原始格式
3. Markdown 引用应使用相对路径，例如 `./media/image_001.png`
4. 图片 alt 文本一期可默认使用文件名

### 6.6.2 图片失败降级规则

1. 单张图片抽取失败时，不中断整篇文档转换
2. Markdown 中应保留失败占位提示，格式可为注释或文本提示
3. 日志中必须记录失败图片位置或编号
4. 至少保证文本和表格结果可输出

---

### 6.7 批量处理需求

系统应支持目录批量转换。

系统应支持递归扫描子目录。

系统应按源文档分别输出对应结果目录。

批量处理时，单个文件失败不应导致全部任务中断。

### 6.7.1 批量日志要求

系统应输出可读的处理日志，至少包含：

- 当前处理文件
- 是否成功
- 输出路径
- 失败原因
- 是否存在降级处理

### 6.7.2 批量汇总要求

批量任务结束后，应输出汇总信息，至少包含：

- 总文件数
- 成功数
- 部分成功数
- 失败数

---

### 6.8 命令行使用需求

系统应提供统一命令行入口。

系统应支持以下核心参数：

| 参数 | 是否必填 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `--input` | 是 | 无 | 输入文件或目录路径 |
| `--output` | 是 | 无 | 输出根目录 |
| `--split` | 否 | `false` | 是否按标题拆分 |
| `--split-level` | 否 | `1` | 拆分层级，仅在 `--split=true` 时生效 |
| `--recursive` | 否 | `false` | 输入为目录时是否递归扫描 |
| `--extract-images` | 否 | `true` | 是否抽取图片 |
| `--log-file` | 否 | 无 | 可选，输出日志到文本文件 |
| `--doc-converter` | 否 | `auto` | `.doc` 转换方式，可选 `auto`、`word`、`libreoffice` |

### 6.8.1 参数规则

1. `--input` 和 `--output` 为必填参数
2. `--split-level` 仅允许取值 `1` 或 `2`
3. 当 `--split=false` 时，传入 `--split-level` 不报错，但应提示该参数未生效
4. 当输入是单个文件时，`--recursive` 参数应被忽略并提示
5. 当 `--extract-images=false` 时，不生成 `media/` 目录
6. 当输入中包含 `.doc` 文件时，系统应自动检查可用转换器
7. `--doc-converter=word` 时，应在日志中提示依赖 Microsoft Word
8. `--doc-converter=libreoffice` 时，应在日志中提示依赖 LibreOffice / soffice

### 6.8.2 命令示例

```bash
word-to-markdown --input "./docs/demo.docx" --output "./output"
word-to-markdown --input "./docs" --output "./output" --recursive --split --split-level 2
word-to-markdown --input "./docs/demo.docx" --output "./output" --extract-images false
word-to-markdown --input "./legacy-docs" --output "./output" --recursive --doc-converter auto
```

### 6.8.3 退出码约定

- `0`：全部成功
- `1`：任务完成，但存在部分失败或降级处理
- `2`：输入参数错误或输入路径非法
- `3`：程序运行异常，未能完成当前任务

当 `.doc` 转换依赖缺失时：

- 单文件模式下，若输入为 `.doc`，应返回 `2` 或 `3`，并明确提示缺少转换依赖
- 批量模式下，缺失依赖导致部分 `.doc` 失败而其他 `.docx` 成功时，应返回 `1`

### 6.8.4 Windows 使用要求

系统应支持 Windows 环境下的一键运行脚本。

脚本应满足：

1. 非开发人员可按说明直接执行
2. 脚本支持传入输入路径和输出路径
3. 脚本运行失败时能看到明确错误信息

## 7. 非功能需求

### 7.1 准确性

优先保证章节、段落、表格、图片的结构正确。

不以 Word 样式完全还原为目标。

应明确复杂文档存在一定误差空间，避免用户错误预期。

### 7.1.1 一期准确性底线

- 标准 Heading 样式标题识别准确率应达到 100%
- 普通段落与标题的相对顺序不得错乱
- 基础二维表格不允许出现整表丢失
- PNG、JPG、JPEG 常见嵌入图片不允许无提示丢失
- `.doc` 预转换成功后，其结构化输出验收标准应与原生 `.docx` 保持一致

### 7.2 稳定性

单个文档解析失败时，应给出清晰错误提示。

批量处理时，应保证部分失败不影响整体任务继续执行。

对缺失图片、特殊表格、异常样式等情况，应尽量降级而非整体崩溃。

### 7.3 可维护性

系统结构应具备清晰边界，至少区分：

- 输入层
- 解析层
- Markdown 输出层
- 拆分层
- 图片处理层
- 日志与任务汇总层

对外应只保留一个推荐稳定入口，避免多版本入口导致混乱。

### 7.4 可扩展性

系统应支持后续能力扩展，包括但不限于：

1. 更复杂表格处理
2. 更精细的图片位置还原
3. 图注识别
4. front matter / metadata 输出
5. 知识库切分策略
6. 自动化测试与样例库建设

### 8.2 输出

- `full.md`
- `sections/` 目录

## 9. 边界与限制

系统当前优先面向 `.docx`，并兼容 `.doc` 输入。

其中 `.doc` 兼容能力依赖外部转换器，不承诺脱离转换依赖进行原生解析。

对于以下场景，允许一期存在不完全准确：

- 非标准标题识别
- 复杂表格
- 图片精确位置还原
- Word 样式保真

项目目标是结构化可用，而不是视觉完全还原。

---

## 10. 一期范围

一期必须覆盖：

1. 单文件与批量 `.docx` / `.doc` 处理
2. 段落、标题、列表转换
3. 基础表格转换
4. 按标题拆分
5. 图片抽取与 Markdown 引用
6. 命令行运行
7. Windows 一键执行
8. 失败不中断与日志输出

---

## 11. 二期规划

二期可考虑：

1. 复杂表格增强
2. 图注识别
3. front matter 输出
4. 更细粒度章节切分
5. 脱离外部依赖的 `.doc` 原生解析可行性评估
6. 与知识库 / 向量库流程对接
7. 真实样例集成测试

## 12. 验收标准

### 12.1 验收样例集

一期验收至少准备以下样例文档：

1. 样例 A：标准 Heading 文档，含 3 级标题、普通段落、列表
2. 样例 B：非标准标题文档，含 `一、`、`（一）`、`1.1`
3. 样例 C：基础二维表格文档，含单元格换行
4. 样例 D：含 PNG/JPG 图片文档，图片穿插在正文中
5. 样例 E：批量目录样例，至少含 10 个 `.docx` / `.doc`，其中 1 个故意损坏
6. 样例 F：单个 `.doc` 文档，验证自动预转换与后续解析链路

### 12.2 基础验收

给定样例 A，系统必须：

- 成功生成 `full.md`
- 标准 Heading 标题识别正确率达到 100%
- 列表输出为合法 Markdown 列表
- 文本段落顺序与原文一致

给定样例 B，系统必须：

- 能识别文档中约定范围内的非标准标题
- 识别错误的标题数量不超过 10%
- 未识别的疑似标题不得导致整篇结构错乱

给定样例 C，系统必须：

- 成功输出基础 Markdown 表格
- 表头和数据行数量与原文一致
- 单元格换行文本不丢失

给定样例 D，系统必须：

- 成功抽取全部 PNG/JPG/JPEG 图片
- Markdown 中图片引用数量与成功抽取数量一致
- 图片插入顺序与原文相对顺序一致

给定样例 F，系统必须：

- 在具备转换依赖的环境下成功将 `.doc` 转换并生成 `full.md`
- 转换后的标题、段落、表格、图片处理规则与 `.docx` 保持一致
- 转换过程失败时给出明确错误信息

### 12.3 拆分验收

给定含一级和二级标题的文档，系统开启拆分后必须：

- 同时生成 `full.md` 和 `sections/`
- `split-level=1` 时按一级标题生成章节文件
- `split-level=2` 时按二级标题生成章节文件
- 章节文件顺序与原文顺序一致
- 文件命名不因特殊字符写入失败

### 12.4 批量验收

给定样例 E，系统必须：

- 逐个处理文件
- 为每个成功文件生成对应输出目录
- 损坏文件报错但不阻塞其他文件继续执行
- 任务结束后输出总数、成功数、部分成功数、失败数
- 最终退出码为 `1`

### 12.5 使用验收

非开发人员按照说明文档，能够在 Windows 环境下完成一次完整转换，且：

- 能成功指定输入路径和输出路径
- 能看懂失败提示
- 能在输出目录中找到 `full.md`、`sections/`、`media/`
